# Path to 95% Accuracy — Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Raise combined streaming accuracy from ~65% to 95%+ across both test corpuses (96 samples).

**Architecture:** Two parallel workstreams: (A) algorithm improvements exploiting validated CTC acoustic hypothesis — the correct verse wins acoustically even when greedy decode produces garbage; (B) model fine-tuning with LoRA for the remaining hard cases. The CTC rescore (`scoreCtcSequence`) already exists and is proven to rank correct verses higher; we just need to use it for candidate SELECTION, not just margin gating.

**Tech Stack:** TypeScript, ctc-rescore.ts (scoreCtcSequence, scoreCtcCandidates), tracker.ts, quran-db.ts, types.ts

**Validated Hypothesis:** CTC forward probability for correct verse beats wrong verse in 4/4 retasy failures tested (margins: 0.38-1.43 nats/token). See `test/ctc-hypothesis.ts`.

### Task 1: Acoustic-first candidate selection in discovery

Currently, discovery uses text matching (Levenshtein ratio) to find the best verse, and CTC acoustic scoring only for margin gating. When the greedy CTC decode is wrong, the text match picks the wrong verse even though acoustic evidence favors the correct one.

**Fix:** After CTC rescoring candidates, if the acoustic-best candidate is clearly better than the text-best AND text confidence is low, override with the acoustic winner.

- `ACOUSTIC_OVERRIDE_TEXT_THRESHOLD`: when text-best score is below this, acoustic override is considered

    const acousticBest = ranked.find((r) => r.feasible);
    if (match) {

      const rescoredMatch = ranked.find(

        match.score < ACOUSTIC_OVERRIDE_TEXT_THRESHOLD &&
        acousticBest.acousticMargin >= ACOUSTIC_OVERRIDE_MIN_MARGIN

            ayah: acousticBest.candidate.ayah,
            ayah_end: acousticBest.candidate.ayah_end,

            score: Math.max(match.score, 0.5), // synthesize confidence
            raw_score: acousticBest.candidate.raw_score,
            bonus: acousticBest.candidate.bonus,
          };
          acousticMargin = acousticBest.acousticMargin;
          lengthFit = acousticBest.lengthFit;

      }

### Task 3: Multi-verse prefix scoring in Pass 1

When a multi-verse transcript (e.g., 20 words from surahs 113:1-5) is compared against individual verses (e.g., 4-word verse 113:1), ratio() gives ~0.3 because the transcript is much longer. The correct surah never enters the top-20 for Pass 2 span matching.

**Fix:** In retrieveCandidates, also compute ratio of text PREFIX (first N words, where N = verse word count) against each verse. This lets short first-verses score correctly.

## Tuning Notes

- `ACOUSTIC_OVERRIDE_TEXT_THRESHOLD = 0.55`: Triggers acoustic override when text match is below this. The retasy failures have text scores of 0.3-0.5, so 0.55 captures them. Don't go higher than 0.65 or it may override correct text matches.
- `ACOUSTIC_OVERRIDE_MIN_MARGIN = 0.25`: CTC scores in hypothesis test showed margins of 0.38-1.43. 0.25 is conservative. Can lower to 0.15 if needed but risks false overrides.
- `DISCOVERY_EXPANDED_CANDIDATES = 200`: 200 CTC forward passes ≈ 50-100ms for short clips. Acceptable latency. If correct verse still not in set, increase to 500.
- `DISCOVERY_MAX_WINDOW_SECONDS = 60.0`: Doubles the discovery window. May increase memory usage. Monitor inference latency — 60s of audio at 16kHz = 960K samples ≈ 3.6MB.

### Key Findings

1. **Algorithm ceiling is ~78-83% with current model.** ~20/96 samples produce CTC phonemes that don't match the correct verse at all. No text/acoustic matching algorithm can fix this.

2. **95% requires model fine-tuning (Task 6).** The remaining 18+ model-quality failures need the CTC model to produce better phonemes for diverse speakers and short clips.

3. **What worked:** Acoustic override (+6 net), expanded candidates (broader CTC pool), finalFlush commit gate (safe single-cycle commit on silence).

4. **What didn't work:** Prefix scoring (pollutes ranking), larger window (cascade amplification), lowered thresholds (false positives), softened thresholds for repeated leaders (wrong verse can be consistently top-match).
